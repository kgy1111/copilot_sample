# Rust 自作プロトコル通信ライブラリ 仕様書

## 1. 概要

Rust で自作バイナリプロトコルに対応したクライアント側通信ライブラリを作成する。
非同期ランタイムは `tokio` を使用し、USB および TCP/IP の両トランスポートに対応する。

---

## 2. 自作通信プロトコル仕様

### クライアント → サーバー（リクエストフレーム）

| フィールド       | サイズ   | 説明                          |
|----------------|--------|-------------------------------|
| デバイスID       | 2バイト  | 装置種別を識別するID             |
| コマンドID       | 2バイト  | 装置毎に定義されたコマンド         |
| ペイロードサイズ   | 4バイト  | ペイロードのバイト数（0の場合あり） |
| ペイロード       | 可変長   | コマンドに応じたデータ            |

### サーバー → クライアント（レスポンスフレーム）

| フィールド       | サイズ   | 説明                          |
|----------------|--------|-------------------------------|
| デバイスID       | 2バイト  | 装置種別を識別するID             |
| アンサーコード    | 2バイト  | 応答コード（例: 0x0000 = OK）   |
| ペイロードサイズ   | 4バイト  | ペイロードのバイト数（0の場合あり） |
| ペイロード       | 可変長   | 応答データ                      |

### 使用例

```
装置種別A: デバイスID = 0xAAAA
装置種別B: デバイスID = 0xBBBB
装置情報取得コマンド: 0x1111（ペイロードなし → ペイロードサイズ = 0）
```

> **ペイロードサイズ上限**: 実装はペイロードサイズの上限を `MAX_PAYLOAD_SIZE = 16 MiB` としている。これを超えるサイズのエンコード・デコード時は `ProtocolError::DecodeError` / `EncodeError` を返す。

## 3. プロジェクト構成（Cargo ワークスペース）

```
myprotocol/
├── Cargo.toml                      # ワークスペース定義
├── myprotocol-core/                # コア定義クレート
│   ├── Cargo.toml
│   └── src/
│       ├── lib.rs
│       ├── frame.rs                # RequestFrame / ResponseFrame（エンコード/デコード）
│       ├── error.rs                # エラー型（thiserror）
│       ├── config.rs               # 共通 TransportConfig（timeout, retry_count, close_timeout）
│       └── transport.rs            # （空ファイル: Transport トレイトは廃止済み）
├── myprotocol-tcp/                 # TCP/IP トランスポートクレート
│   ├── Cargo.toml
│   └── src/
│       ├── lib.rs
│       ├── config.rs               # TcpConfig（address, port, timeout, retry_count, close_timeout, tls）
│       └── transport.rs            # TcpTransport（tokio::net::TcpStream / TlsStream）
├── myprotocol-usb/                 # USB トランスポートクレート
│   ├── Cargo.toml
│   └── src/
│       ├── lib.rs
│       ├── config.rs               # UsbConfig（vid, pid, timeout, retry_count, close_timeout）
│       └── transport.rs            # UsbTransport（rusb + spawn_blocking）
└── myprotocol-client/              # 統合クライアントクレート
    ├── Cargo.toml
    └── src/
        └── lib.rs                  # Client（enum TransportKind による静的ディスパッチ）
```

### ディスパッチ方式

`Client` 内部では `enum TransportKind { Tcp(TcpTransport), Usb(UsbTransport) }` による
静的ディスパッチを採用している（`Box<dyn Transport>` + `async_trait` による動的ディスパッチは廃止済み）。
`TcpTransport` / `UsbTransport` はトレイト実装ではなく、それぞれの `impl` ブロックに
`connect` / `send` / `close` メソッドを直接定義する。

## 4. コンフィグ仕様

### 共通設定

```rust
pub struct TransportConfig {
    pub timeout: Duration,        // 通信タイムアウト（デフォルト: 30s）
    pub retry_count: u32,         // 再送回数（デフォルト: 0）
    pub close_timeout: Duration,  // close() 時の最大待機時間（デフォルト: 5s）
}
```

### TcpConfig

```rust
pub struct TcpConfig {
    pub address: String,          // IPアドレス（設定ファイルから注入）
    pub port: u16,                // ポート番号
    pub timeout: Duration,        // デフォルト: 30s
    pub retry_count: u32,         // デフォルト: 0
    pub close_timeout: Duration,  // デフォルト: 5s
    pub tls: Option<TlsConfig>,   // None = 平文TCP、Some = TLS有効
}

pub struct TlsConfig {
    pub server_name: String,                   // SNI ホスト名（証明書検証に使用）
    pub ca_cert_pem: Option<Vec<u8>>,          // カスタムCA証明書（None = システム証明書）
    pub client_cert: Option<ClientCertConfig>, // クライアント証明書（mTLS用）
    pub danger_accept_invalid_certs: bool,     // 自己署名証明書許可（⚠ 本番禁止）
}

pub struct ClientCertConfig {
    pub cert_pem: Vec<u8>,  // クライアント証明書 PEM バイト列
    pub key_pem:  Vec<u8>,  // 秘密鍵 PEM バイト列（PKCS#8 または PKCS#1）
}
```

### UsbConfig

```rust
pub struct UsbConfig {
    pub vid: u16,                 // ベンダーID（後から注入・フィルタリングに使用）
    pub pid: u16,                 // プロダクトID（後から注入・フィルタリングに使用）
    pub timeout: Duration,        // デフォルト: 30s
    pub retry_count: u32,         // デフォルト: 0
    pub close_timeout: Duration,  // デフォルト: 5s
}
```

## 5. 主要 API

```rust
// TCP クライアント（平文）
let mut client = Client::new_tcp(TcpConfig {
    address: "192.168.1.100".to_string(),
    port: 8080,
    tls: None,
    ..Default::default()
});
client.connect().await?;
let resp = client.send(0xAAAA, 0x1111, vec![]).await?;
println!("answer: {:#06x}, payload: {:?}", resp.answer_code, resp.payload);
client.close().await?;

// TCP クライアント（TLS有効）
let mut client = Client::new_tcp(TcpConfig {
    address: "192.168.1.100".to_string(),
    port: 8443,
    tls: Some(TlsConfig {
        server_name: "mydevice.local".to_string(),
        ca_cert_pem: Some(include_bytes!("ca.pem").to_vec()),
        client_cert: None,
        danger_accept_invalid_certs: false,
    }),
    ..Default::default()
});

// USB クライアント（VID/PID は後から注入）
let mut client = Client::new_usb(UsbConfig {
    vid: 0x1234,
    pid: 0x5678,
    ..Default::default()
});
client.connect().await?;  // connect() 内でエンドポイントバッファをフラッシュ
let resp = client.send(0xBBBB, 0x1111, vec![]).await?;
client.close().await?;
```

## 6. close() ロジック仕様

### 挙動（案B: close専用タイムアウト方式）

```
close() 呼び出し
  │
  ├─ 通信中でない → 即時クローズ
  │
  ├─ 送受信実行中 → 完了を close_timeout 以内で待機
  │                    完了 → 正常クローズ
  │                    タイムアウト → 強制切断
  │
  └─ タイムアウト待ち中 → タイムアウト満了 or close_timeout のどちらか早い方で切断
```

- `close_timeout` のデフォルト値: **5秒**
- `close_timeout` は TCP/USB コンフィグそれぞれで個別設定可能

## 7. TLS/SSL 設計方針

| 状態 | 内部実装 |
|---|---|
| `tls: None`（無効） | `tokio::net::TcpStream` をそのまま使用 |
| `tls: Some(...)`（有効） | `TlsConnector::connect()` → `TlsStream<TcpStream>` |

- 内部で `enum BoxedStream { Plain(...), Tls(...) }` に統一し、**外部 API は変更なし**
- `pin-project` で `AsyncRead` / `AsyncWrite` をゼロコストで委譲
- `danger_accept_invalid_certs: true` の場合も、`client_cert` が設定されていれば mTLS は有効になる
- USB はハードウェア物理接続のため TLS は対象外
- TLS ライブラリ: `tokio-rustls` + `rustls`（pure Rust、クロスプラットフォーム）

---

## 8. USB 設計方針

- VID/PID は後から注入でき、複数デバイス接続時にフィルタリングして選択可能
- `connect()` 時に USB エンドポイントバッファをフラッシュ（再接続時の古いデータ受信防止）
- **ハンドシェイク・認証ロジックはライブラリ側に持たない**（装置毎に異なるため上位アプリに委譲）
- `connect()` 内の全 USB 操作（`open_device_with_vid_pid` / `claim_interface` / バッファフラッシュ）は `spawn_blocking` 内で実行し、tokio スレッドをブロックしない
- USB Bulk 転送は 1 回の呼び出しで要求バイト数を保証しないため、`read_bulk_exact` ヘルパーで全バイト到達までループ読み取りを行う
- Bulk IN / OUT エンドポイントは **同一インターフェース記述子** から取得する（異なる IF の混在防止）

```rust
// 上位アプリ側で装置毎に自由に実装する例
client.connect().await?;
// 装置A: コマンドID 0x0001 で疎通確認
let resp = client.send(0xAAAA, 0x0001, vec![]).await?;
// 装置B: コマンドID 0x0010 で疎通確認
let resp = client.send(0xBBBB, 0x0010, vec![]).await?;
```

## 9. 依存クレート一覧

| クレート | 用途 | 対象クレート |
|---|---|---|
| `tokio` (full) | 非同期ランタイム | 全体 |
| `thiserror` | エラー型定義 | core |
| `rusb` | USB libusb バインディング（Windows対応） | usb |
| `tokio-rustls` | Tokio + rustls 統合 | tcp |
| `rustls` | TLS 実装本体 | tcp |
| `rustls-native-certs` | システム証明書チェーン読み込み | tcp |
| `rustls-pemfile` | PEM 証明書ファイル解析 | tcp |
| `pin-project` | enum の `AsyncRead`/`AsyncWrite` ゼロコスト実装 | tcp |
